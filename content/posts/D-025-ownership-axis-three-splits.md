# Три расщепления одного понятия: как «ось владения» распутывает перегруженные термины

## Тезис

Когда термин перегружен («цифровой двойник», «база данных», «IWE»), инстинкт — искать подтермины, делить на semantic / episodic, fast / slow, user-facing / internal. Это не работает. Распутывает только **ось владения**: кто пишет, кто владеет, где source-of-truth.

За один день 22 апреля 2026 я применил этот приём три раза к трём разным понятиям. Каждый раз аморфное расплылось в 3-4 чётких слоя, и каждый раз граница прошла ровно по writer+owner, а не по тому, где я её искал сначала.

## Расщепление 1. «Цифровой двойник» → Персона / Память / Контекст

Долгое время «ЦД» был монолитом: всё, что платформа знает о пользователе. В этом монолите жили вперемешку: декларации пользователя о себе, события от платформы, промпт для LLM-вызова.

Первая попытка разделить — по когнитивной аналогии (semantic / episodic memory, как у Letta, Mem0, LangMem). Не получилось: одни и те же таблицы попадали в разные слои в зависимости от трактовки.

**Ось владения сработала:**

- **Персона** — writer = пользователь, owner = его Git-репо (PACK-personal, DS-my-strategy, preferences). Платформа — read-only.
- **Память** — writer = платформа автоматически, owner = Neon. События, платежи, расчётный baseline.
- **Контекст** — writer = агент в runtime, owner = runtime (не хранится). Промпт-сборка под один LLM-вызов.

Тест границы: «Что пропадёт, если удалю X?» — Git пользователя → Персона. Neon → Память. Прервать LLM-вызов → Контекст.

## Расщепление 2. «База данных платформы» → 12 БД по OwnerIntegrity

Второй случай — техническая карта данных. В старой архитектуре было 9 баз, но границы между ними были проведены по удобству (где исторически лежала таблица), а не по владению фактом.

Инстинктивный вопрос: «как делить БД — по домену? по BC? по сервису?» — давал разные ответы, и все были плохие. В каждом варианте какой-то класс фактов оказывался в двух местах одновременно.

**Ось владения — тот же приём:** для каждой сущности спрашиваешь не «где её удобно хранить», а «кто её пишет». Если писатель один — сущность в одной БД. Если писателей несколько — это сигнал, что сущность на самом деле составная.

Карта разложилась на 12 БД. Три новые возникли не из feature-требования, а из того, что у трёх классов фактов обнаружился отдельный writer: health-данные (БД #8), knowledge-индексы (отдельно от knowledge-контента), learning-объекты (отдельно от aist-bot).

Ключевое наблюдение: **факт принадлежит не системе, которая им пользуется, а системе, которая его производит.** БД #8 health владеет данными о здоровье ВСЕХ систем — не каждая система хранит собственный health.

## Расщепление 3. «IWE как продукт» → L2 общая платформа + L3 персональный Git

Третий случай — продуктовая граница open-core. Где проходит граница между открытой бесплатной частью и подписочной?

Типичный ответ — по фичам: basic open-source, premium в подписке. Но IWE про интеллектуальную работу, а не про фичи. Feature-gate ломает саму идею экзокортекса: если часть моего мышления живёт в подписке, отключение подписки отнимает мою же память.

**Ось владения даёт другой ответ:** граница проходит не по функциям, а по месту владения данными.

- **L2 — общая платформа.** Живёт у поставщика. Общие для команды знания, MCP-сервисы, модели. Меняется платформой, потребляется всеми.
- **L3 — персональный Git пользователя.** Живёт у пользователя. Pack, captures, preferences, DS-my-strategy. Меняется пользователем, никто кроме него не читает/пишет.

Подписка оплачивает доступ к L2. Отключаешь подписку — теряешь L2, но L3 остаётся целиком твоим. Это не вопрос бизнес-модели, это вопрос границы владения. Экзокортекс без этого свойства перестаёт быть экзокортексом.

## Приём в сухом остатке

Когда термин перегружен, **не делите его по семантике** — ищите ось владения:

1. **Кто пишет?** (writer)
2. **Где source-of-truth?** (owner)
3. **Что пропадёт, если удалить носитель?**

Три ответа задают три слоя. Границы между слоями проходят ровно там, где меняется хотя бы один ответ.

Это не универсальный закон онтологии, это рабочий приём. Он сработал три раза за день на разных по масштабу понятиях — от структуры одного пользовательского профиля до границы коммерческого продукта. Когда почувствуешь, что термин «расплывается» и подтермины не держатся — попробуй.
