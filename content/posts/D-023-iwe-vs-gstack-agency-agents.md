# Три подхода к систематизации ИИ-работы: gstack, Agency Agents и IWE

## 1. Крючок

Март 2026. CEO Y Combinator публикует на GitHub свою личную систему работы с Claude Code.

31 TypeScript-скилл. Виртуальная команда из ролей: CEO review, Engineering review, Design review. Артефактная цепочка от мысли до деплоя.

57 000 звёзд за три недели.

Это не продукт компании. Не стартап. Личная система конкретного человека — Гэрри Тана. Люди форкают её так быстро, что GitHub тормозит.

Пятью месяцами раньше та же история с Agency Agents — 100+ Markdown-карточек агентов-специалистов для Claude Code, собранных независимым разработчиком Марком Ситаржевским. 66 000 звёзд. Сообщество присылает PR с новыми агентами — от WeChat Mini Program Developer до Government Digital Presales.

Что происходит? Люди нашли инструмент (Claude Code, Cursor, Windsurf) и поняли: он умеет. Но непонятно, как правильно. Нет структуры. Каждый изобретает свою.

Эти два проекта дали ответ — конкретный, с низким порогом входа. Поэтому вирус.

Я смотрю на них несколько месяцев. Использую IWE — свою систему того же класса. В этой статье — не реклама и не сравнительный тест. Разбор того, какую *архитектурную ставку* делает каждый подход. Это важно, потому что выбор не очевиден.

---

## 2. Три системы — кратко

**gstack** — пайплайн спринта.

Модель простая: каждый этап работы — роль, каждая роль производит артефакт для следующей. `/investigate` не начинает фикс без диагностики. `/design-review` оценивает по шкале 0–10 и явно проверяет на «AI Slop» — шаблонность без мысли. Кросс-модельный ревью: та же задача уходит в OpenAI Codex, расхождения сравниваются. Параллельные спринты масштабируют объём.

Архитектурная ставка: *дисциплина через артефакты*.

**Agency Agents** — каталог специалистов с характером.

Каждый агент — Markdown-карточка с одним полем `vibe`:

> `vibe: "I ship first, optimize later. Done > perfect."`

Одна строка задаёт суть и предотвращает дрейф. Конвертер позволяет использовать те же карточки в Cursor, Aider, Windsurf — не привязан к одному инструменту. Охват широкий: что-то найдётся для любой задачи.

Архитектурная ставка: *низкий порог + personality как anti-drift*.

**IWE** — дистрибутив с версионированием.

Не набор скиллов — операционная среда. Три слоя: платформа (методология + механика), расширения пользователя, авторские правила. Обновление через `update.sh` с 3-way merge: твои правки в §9 не затираются при выходе новой версии платформы. Протоколы с extension points: добавить шаг в Day Close — один файл, без правки платформенных файлов.

Архитектурная ставка: *среда, которая обновляется без потери кастомизаций*.

## 3. Живой пример: одна задача — три реакции

Возьмём конкретный момент. Не абстрактный сценарий — реальный.

Несколько дней назад я разбирался, почему pg_dump падает при подключении к Neon через стандартный connection string. Два часа. Причина — pooled endpoint работает в transaction mode, и pg_dump, который держит транзакцию открытой, просто не совместим с этим режимом. Решение: для миграций всегда использовать unpooled endpoint.

Это маленькое, но дорогостоящее знание. Дорогостоящее, потому что следующий раз, когда кто-то (или я сам в другом контексте) наступит на это — снова два часа.

Смотрим, что происходит с этим знанием в каждой системе.

---

**gstack**

Сессия прошла отлично. `/investigate` дал структурированную диагностику, `/build` зафиксировал фикс, `/review` проверил. Артефактная цепочка сработала — внутри сессии.

После `/ship` — тишина. Знание о pooled/unpooled живёт в commit message. Если написал подробно — повезло. Если нет — в следующий раз Claude предложит тот же сломанный путь, потому что у него нет памяти между сессиями.

gstack не проектировался для хранения знаний. Это не упрёк — это архитектурная ставка.

---

**Agency Agents**

Ты выбираешь `DevOps Engineer` с `vibe: "I automate everything that runs more than twice"`. Агент полезен внутри задачи. Но карточка статична — она не знает, что ты уже проверял pooled endpoint и он не работает.

Можно добавить в карточку кастомное правило. Но это ручная работа, не механика системы. Следующее обновление Agency Agents перетрёт твою карточку, если ты её форкнул, или не обновится, если не форкнул.

Связи между сессиями нет.

---

**IWE**

Конец сессии — рубеж. Протокол Работы задаёт один вопрос: *есть ли знание для записи?*

Я говорю: «Capture: pooled endpoint ломает pg_dump — hard-distinction».

Три строки в `memory/reference_neon_connections.md`:

```
Pooled endpoint ≠ unpooled endpoint (Neon).
Pooler = transaction mode — pg_dump, LOCK, BEGIN-COMMIT падают.
Миграции → unpooled. Long-running сервисы → pooled.
```

Всё. Следующая сессия — Claude читает этот файл в начале и не предлагает сломанный путь. Не потому что умнее, а потому что система передала знание.

Но это ещё не самое интересное.

Дальше это различение попадает в `distinctions.md` — общий реестр. И если в Pack (корпус доменного знания) есть документы про Neon, система предложит propagate туда же. Одно наблюдение из одной сессии становится частью структурированного корпуса знаний.

---

Сравним в одной таблице:

| | gstack | Agency Agents | IWE |
|---|---|---|---|
| Знание внутри сессии | ✅ артефактная цепочка | ✅ агент с характером | ✅ протокол работы |
| Знание между сессиями | ❌ только git history | ❌ статичные карточки | ✅ memory + capture |
| Обновление без потерь | ❌ ручной merge | ❌/⚠️ форк или потеря | ✅ 3-way merge |
| Накопление корпуса | ❌ | ❌ | ✅ Pack |

## 4. Главная развилка

Почему gstack и Agency Agents набрали суммарно 123 000 звёзд, а не что-то сложнее?

Потому что они решают реальную проблему прямо сейчас. Ты устанавливаешь скиллы — и работаешь быстрее. Немедленная отдача. Это честно.

IWE требует больше: понять, что такое Pack, зачем протоколы, как работает ОРЗ-фрактал (ритм сессия/день/неделя). Это порог.

Но это тот же порог, что у любого профессионального инструмента. IntelliJ IDEA сложнее Notepad. Git сложнее «сохранить файл». Сложность там — не баг, а встроенная теория о том, как правильно работать с кодом.

Разница в том, *что* каждая система оптимизирует:

- **gstack и Agency Agents** — ускоряют существующую работу. Берёшь то, что делал, делаешь быстрее и структурированнее.
- **IWE** — меняет структуру, в которой работа существует. Не скорость на той же дороге, а другая дорога.

Это не значит, что IWE лучше. Это значит, что они отвечают на разные вопросы. Если тебе нужно ускориться прямо сейчас — gstack. Если тебе важно, чтобы знания накапливались и система обновлялась без потерь — IWE.

Выбор архитектурной ставки зависит от горизонта.

## 5. Что можно взять у конкурентов

Анализ чужих систем — не только про критику. Там есть хорошие идеи.

**Шкала 0–10 из gstack.** При ревью артефакта — явная метрика, не «ок/не ок». Плюс конкретный критерий «AI Slop» — проверка, нет ли шаблонного мышления без содержания. Применимо к верификации знания в Pack: не «добавлено», а «оценено на 7/10, вот почему».

**`vibe`-поле из Agency Agents.** Одна строка — суть роли. Предотвращает дрейф поведения между сессиями. В IWE роли описаны подробнее, но `vibe` как быстрый anti-drift маркер — идея стоящая. Применимо к шаблонам ролей.

**Публичные числа из обоих.** «600k строк за 60 дней» — верифицируемо и убедительно. IWE умеет считать мультипликатор (соотношение задокументированных часов к WakaTime), но не публикует. Метрики результата методологии — не инструмента — это то, чего не хватает для внешней аудитории.

## 6. Три вопроса к своей системе

Если у тебя уже есть система работы с ИИ — набор промптов, скиллов, правил — вот три вопроса, которые стоит задать.

**Как ты обновляешь её без потери правок?**
Если ответ «вручную» или «форк» — у тебя нет механики обновления, есть просто файлы. Это работает до первого большого обновления Claude Code.

**Где хранятся решения, а не только их результаты?**
Код в репо — результат. ADR, который объясняет *почему* именно так — решение. Через шесть месяцев важно именно второе. Если нет явного места для решений — они в головах или потеряны.

**Что происходит с инсайтами из работы через три месяца?**
Если ответ «не знаю» — каждая сессия начинается с нуля. Это не плохо само по себе, но это выбор, который стоит сделать осознанно.

Все три системы — gstack, Agency Agents, IWE — это попытки ответить на эти вопросы. Просто с разными приоритетами.
