# Продуктивность ×2 — это не про инструменты. Это про культуру работы

## Тезис

Программа личного развития — это не список курсов и не расписание. Это **культура работы** с тремя разрезами: протоколы (формализованные последовательности), навыки (нарабатываемые по ситуации), форматы (стандарты оформления). 12 взаимосвязанных элементов. Убери один — посыпется. Добавь лишний — перегрузишь. Ниже — скелет, который я выстроил за год в IWE (Intellectual Work Environment) — операционной системе интеллектуальной работы.

## 1. ОРЗ-фрактал (Открытие → Работа → Закрытие)

Центральный ритм. Применяется фрактально: Сессия, День, Неделя.

**Открытие** — не «сел и начал», а: выбрал роль → выбрал метод → определил рабочий продукт (РП). Без этого шага работа незапланированная — значит, не учтённая, не оценённая, потерянная.

**Работа** — выполнение с захватом знания (Capture-to-Pack) на каждом рубеже.

**Закрытие** — фиксация результата: что сделано, что узнал, куда записал. Пропуск = потеря знания.

Масштаб фрактала:
- **Сессия** (25-90 мин): роль → метод → РП → работа → capture → close
- **День**: Day Open (план + календарь) → работа → Day Close (итоги + governance)
- **Неделя**: WeekPlan → работа → Week Close (ревью + мультипликатор)

## 2. Каскад планирования (Стратегия → Месяц → Неделя → День)

Стратегия (куда иду) → Приоритеты месяца (R1-R{N}, результаты) → План РП на неделю (бюджет в часах) → План РП на день (что именно сегодня).

Каждый уровень сужает фокус. Стратегия без дневного плана — мечта. Дневной план без стратегии — суета.

**Priority Gate:** РП ≥3h → «К какому результату месяца это ведёт?» Нет ответа = off-plan.

## 3. Рабочий продукт (РП) — единица учёта

РП = артефакт с критериями готовности, не задача и не описание. «Написать документ» — задача. «Спецификация Web App v1 (5 сценариев, модель данных, API endpoints)» — РП.

Каждый РП имеет: номер, бюджет (часы), приоритет, связь с результатом месяца.

## 4. Выбор роли и метода

Перед работой: **кто я сейчас?** (роль) и **как я буду действовать?** (метод/описание метода).

Роли: Стратег, Кодировщик, Исследователь, Автор, Архитектор... Каждая — свой контекст, свои инструменты, свой фокус.

Метод: описание «как делать» (SOP/рецепт). Для повторяющихся действий — зафиксированный. Для новых — выбрать ближайший и адаптировать.

## 5. ВДВ (Выполнение → Данные → Видимость)

При создании новой системы: какие **процессы** (как делать), какие **данные** (что хранить), какая **видимость** (что показать пользователю). Три аспекта, ни один нельзя пропустить.

## 6. Capture-to-Pack (захват знания)

На каждом рубеже (подзадача завершена, паттерн замечен, решение принято) — проверка: есть ли знание для записи?

Доменное → Pack. Реализационное → DS. Правило для всех → CLAUDE.md. Крупный урок → memory.

Без этого правила знание остаётся в голове (или в чате) и теряется.

## 7. АрхГейт (оценка архитектурных решений)

7 характеристик: Эволюционируемость, Масштабируемость, Обучаемость, Генеративность, Скорость, Современность, Безопасность (ЭМОГССБ). Порог ≥8 из 10.

Блокирующее: решение не предлагается, пока не прошло оценку.

## 8. Pre-action Gates (проверки перед действием)

Набор «стоп-кранов»:
- WP Gate: любое задание → сначала протокол Открытия
- UC Gate: реализация → проверить use-case
- MAP Gate: какие сервисы затронуты?
- IntegrationGate: новый инструмент → контур, роли, продукты

## 9. Осторожный мультитаскинг

Параллельная работа сессий (несколько окон Claude Code) — но с правилами:
- Каждая сессия = свой РП, своя роль
- Мультипликатор = Σ бюджетов done-РП / физическое время (WakaTime)
- Цель: мультипликатор 1.5-2.5x (60h физических → 100h РП)
- Больше 3 параллельных сессий = coordination overhead > parallelism gain

## 10. Fallback Chain (иерархия истины)

DS (производные) → Pack (доменное знание) → Base (принципы: SPF → FPF → ZP).

При конфликте — побеждает тот, кто выше. Pack — source-of-truth для домена.

## 11. Self-correction (самоисправление)

Обнаружил расхождение с реальностью → немедленно предложить конкретное исправление. Одна ошибка не повторяется дважды.

## 12. Стоп-лист для РП

Когда прекращать работу: бюджет исчерпан (>150%), блокер >2 недель, гипотеза опровергнута, приоритет упал. Антипаттерн — sunk cost fallacy.

## Как они связаны (скелет)

```
Стратегия → Приоритеты месяца → WeekPlan → DayPlan → Сессия
                                                        ↓
                                                  ОРЗ-фрактал
                                                  ↓         ↓
                                            Роль+Метод   Capture
                                                  ↓         ↓
                                              Pre-action   Pack
                                              Gates
```

Убери ОРЗ — нет ритма. Убери Capture — знание не копится. Убери каскад планирования — нет связи «зачем» с «что делаю сейчас». Убери мультитаскинг — одна сессия за раз, мультипликатор = 1.0.

Это **культура работы** — набор протоколов, навыков и форматов, который работает вместо тебя — пока ты работаешь над содержанием. Часть операционной системы интеллектуальной работы (IWE).

## Свежие подкрепления (практикумы 26-27 апр)

**Должность ≠ Роль (повторено в обеих группах, 26.04 и 27.04):**
Начальник — это **должность**, в которой человек одновременно играет три роли:
1. **Операционный менеджер** — поддерживает текущий конвейер (текучка)
2. **Менеджер оргразвития** — улучшает конвейер (важное)
3. **Лидер** — вдохновляет, держит видение

ОС знаниевого работника — это явное распределение этих трёх ролей внутри одного человека. Без этого либо текучка съедает всё (нет оргразвития), либо вечный реформатор (нет операционки), либо одиночество (нет лидерства даже над собой).

**Я начальник себя:**
> «Я начальник себя. У меня тоже текучка, срочное и важное. Слот саморазвития = моё время на важное.» (27.04)

Это не метафора, это рабочая формулировка для проектирования личной ОС.

## Расширение — ВДВ-конвейер вместо матрицы Эйзенхауэра (27.04)

Матрица Эйзенхауэра — самый известный инструмент управления временем. Делишь дела на 4 квадранта по двум осям: **срочное / не срочное** и **важное / не важное**. Учат во всех бизнес-школах. И **она не работает** — не потому что плохая, а потому что **склеивает в одну плоскость разные онтологические уровни**.

Главная проблема: «срочное» и «важное» — **не две независимые оси**. «Важное» — категория, относящаяся к **развитию системы**. «Срочное» — категория, относящаяся к **восстановлению нормального хода работы**. Это **разные типы работы**, которые матрица механически склеивает.

Пытаясь работать с матрицей честно, ты заходишь в тупик: квадрант «важное и срочное» — весь твой день, потому что **всё, что ты делаешь сейчас**, для тебя так или иначе срочное и важное. А квадрант «не важное и не срочное» — пустой, потому что туда сам не пойдёшь. Матрица не объясняет, **почему дни заполнены текучкой**.

В системном подходе любая работа = **конвейер «Вход → Действие → Выход»**, делится на **три типа**, которые **не пересекаются**:

### 1. Текущее (плановая работа конвейера)
То, для чего конвейер построен. Бухгалтер закрывает месяц. Разработчик пишет код по тикетам. Родитель отводит ребёнка в школу. Конвейер работает по проектному плану. Внимание — на исполнение.

### 2. Срочное (восстановление нормального хода)
Конвейер сломался. Машина не завелась — как ребёнка везти? Прод упал — как поддерживать сервис? Срочное = **возврат к норме**. Не улучшение, а восстановление. Внимание — на минимально необходимое, чтобы вернуть конвейер в строй.

### 3. Важное (улучшение конвейера)
Меняет сам конвейер: новые методы, инструменты, роли, инфраструктура. **Важное не нужно сегодня** — без него конвейер крутится. Но без важного конвейер **не развивается**, и через год отстаёт от мира. Внимание — на проектирование и инвестиции.

### Почему это лучше Эйзенхауэра

ВДВ-конвейер сразу даёт три **разных режима работы**, и режимы не путаются:
- В режиме «текущего» не задумываюсь о методе — он уже выбран.
- В режиме «срочного» не пытаюсь улучшить конвейер — нужно его срочно починить.
- В режиме «важного» не беру в работу мелкое срочное — оно сожрёт время и не даст результата.

Главное: если я ловлю себя на «срочном», и оно повторяется регулярно — это сигнал, что мой конвейер сломан **системно**, и нужно перейти в режим «важного» и его перестроить. Эйзенхауэр такой сигнал не даёт.

### Применение

1. Возьми один день из последней недели.
2. Раздели всё на три ведра: текущее / срочное / важное.
3. Посчитай долю каждого.
4. Если **срочного больше 20%** — конвейер сломан, разбираться **системно** (важное), а не тушить.
5. Если **важного 0%** — конвейер не развивается, через год отстанешь от мира.

Норма: ~70% текущее / ~10% срочное / ~20% важное (меняется по роли).
